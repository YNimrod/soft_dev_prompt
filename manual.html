概要

このプロンプト集は、ソフトウェア開発をAIと協働で進めるための仕組みです。Phase 1（ユーザーストーリー作成）からPhase 6（ドメインモデリング）までの全工程をカバーし、ナビゲーターが毎回適切なフェーズへ誘導します。

🔍 このプロンプト集の強み

基本プロンプトをベースに、手戻り防止・品質向上・知見の蓄積の3軸で大幅に強化しています。

💾 中間成果物の保存
各ステップ完了ごとにファイル保存。どのフェーズからでも再開可能。
🎭 アクターレビュー（Phase 1c）
Phase 1 で定義したアクター自身の視点で行動の自然さ・目的・抜け漏れを確認。
👥 専門家ペルソナレビュー（Phase 1d）
PM・UX・技術・法務・マーケターから使うペルソナを選択可。スキップも可能。
📋 インセプションサマリー
セッションクリア前にナビゲーターが inception_summary.md を生成。Phase 2 以降への引き継ぎを一元管理。
🔗 I/F 手戻り防止
Phase 3〜6 にわたる if_precheck 連鎖でI/F設計の合意を事前形成。
📝 ナレッジログ
各フェーズ完了時に knowledge_log.md へ再利用可能な知見を自動蓄積。
🛑 歯止めルール
1ステップごとに停止・承認待ち。AIのオーバーワーク・先読みを防止。
❓ 強化版 Question
問い・背景・選択肢・影響を必ず明示。判断しやすい質問フォーマット。

開発フロー

Phase 1 から順番に進めます。各フェーズの成果物をファイルに保存してから、セッションをクリアして次のフェーズへ進んでください。

⚠️ セッションクリアのタイミング

各フェーズ完了後、成果物を保存してからセッションをクリアしてください。Phase 1・1b・1c は同一セッションで実施します。Phase 1d はオプション（スキップ可）。セッションクリア前にナビゲーターが inception_summary.md を生成してから Phase 2 へ進みます。

1
Phase 1 — ユーザーストーリー作成
熟練PMの役割で7ステップのユーザーストーリーを作成。ST4開始前にエピック数・想定ストーリー数を提示し、レビュー粒度（[A]エピック単位 / [B]ストーリー単位 / [C]一括）をAIが推奨・ユーザーが選択。
01_generate_user_stories_plan.md
1b
Phase 1b — 整合性チェック
同一セッションで実施。問題をカテゴリ別（矛盾・用語・AC品質・AC粒度・網羅性・申し送り）に一覧提示。AC粒度の問題はSTEP 2.5で具体化案を複数提示。後工程への申し送りは phase1b_handover.md に保存。
01b_check_user_stories_consistency.md
1c
Phase 1c — アクターレビュー ★新設
Phase 1 ST2 で定義したアクターを1人ずつ「行動の自然さ・目的との対応・抜け漏れ」の3観点でレビュー。全アクター完了後にサマリーを保存。
01c_actor_review.md
1d
Phase 1d — 専門家ペルソナレビュー ★オプション
起動時に5ペルソナの概要を提示し使用するペルソナを選択。「スキップ」でフェーズ全体をスキップして Phase 2 へ進める。全Question回答後にセッションクリア前に inception_summary.md を生成。
01d_multi_perspective_review.md
セッションクリア前に inception_summary.md を生成 → セッションクリア
2
Phase 2 — 画面モック作成
STEP 0 で作成物の種類（WEBアプリ/エージェント/その他）・粒度（ワイヤーフレーム/動くモック）・作業順序（フローチャート先行/直接作成）を選択。フローチャート先行時はSTEP 1aで網羅性確認、STEP 1bで未決事項を精査。inception_summary.md を参照して引き継ぎ事項を反映。
02_generate_screen_mock.md
セッションクリア
3
Phase 3 — Unit of Work 分割
STEP 0 で前工程成果物（inception_summary / actor_review / screen_flow）の影響を確認してから計画作成。各ユニットは概要（目的・主な責務・境界の根拠・他ユニットとの連携）を先に承認してからストーリー定義。I/F事前確認はSTEP A/Bの2段階で実施。
03_generate_unit_of_work.md
セッションクリア（if_precheck.md の全 Question 回答済み後）→ Construction へ
4
Phase 4 — コンテキストマップ作成
STEP 1でコンテキストマップ＋連携一覧テーブル、STEP 2でシーケンス図、STEP 3でI/F事前確認を2段階で実施。STEP 1の整合性確認でPACT定義間の依存関係（パターンA/B/C）を識別。
04_generate_context_map.md
セッションクリア（if_precheck_p4.md の全 Question 回答済み後）
5
Phase 5 — PACT 定義
計画作成時に着手順序をAIが提案（未解決事項なし優先・共有ユニット連続着手）。各連携はSTEP A（I/F確認一覧）→STEP B（PACT定義）の2段階。STEP Bの整合性確認でユニットの主な責務の範囲内かを確認。
05_generate_pact.md
セッションクリア（pact_index.md の未解決 Question がゼロになった後）
6
Phase 6 — ドメインモデリング
STEP A（I/F確認）→STEP B（計画）→STEP C（コンポーネント定義）の3段階。関連する全PACTファイルを参照。ACL担当コンポーネントの明示を計画・定義の両段階で確認。ユニットごとに繰り返し実行。
06_generate_domain_model.md
ユニットごとにセッションクリア

ナビゲーターの使い方

ナビゲーターはフェーズの進行を管理するプロンプトです。新しいセッションを開始するたびに、以下のいずれかをClaude Codeに入力してください。

🚀
インセプション用
Phase 1 〜 3
navigator_inception.md
🔨
コンストラクション用
Phase 4 〜 6
navigator_construction.md
🗺️
全フェーズ通し
Phase 1 〜 6
navigator_prompt.md

ナビゲーターへの入力例（インセプション）

以下のファイルを順番に読み込んでください。

1. C:\Users\%USERNAME%\soft_dev_prompt\knowledge.md
2. C:\Users\%USERNAME%\soft_dev_prompt\00_README.md

読み込んだ内容をもとにナビゲーターとして動いてください。

ナビゲーターの役割

  • 現在のフェーズを把握し、次のアクションを案内する
  • 各フェーズの移行前にフェーズ移行チェックリストを提示してユーザーに確認させる
  • Phase 1 ST4 開始前にレビュー粒度（A/B/C）を推奨とともに提案する
  • Phase 1d の開始時に5ペルソナの概要を提示し使用するペルソナを選ばせる（スキップ可）
  • セッションクリア前に inception_summary.md を生成して保存する
  • 各フェーズの md ファイルを自動で読み込んで使用する
  • Question が発生したら強化版フォーマット（問い・背景・選択肢・影響）で記載する
  • 各フェーズ完了時に knowledge_log.md への追記を案内する
  • 独自判断をせず、不明点は必ず確認する
✅ セッション切り替えのタイミング

Phase 1 → 1b → 1c は同一セッション。Phase 1d（オプション）完了またはスキップ後、セッションクリア前に inception_summary.md を生成してから Phase 2 へ。Phase 4→5→6 はコンストラクションナビゲーターを使用。

主な特徴

① レビュー体制の強化（Phase 1c・1d）

Phase 1b の整合性チェック後に2段階のレビューを実施します。

フェーズレビュアー主な確認観点備考
Phase 1c定義したアクター自身行動の自然さ・目的との対応・抜け漏れ必須
Phase 1d5専門家ペルソナPM/UX/ARCH/LEGAL/MKT 各視点の懸念オプション・スキップ可

Phase 1d の5ペルソナは起動時に選択できます。不要なペルソナは外せます。

ペルソナ略称主な確認観点
📦 プロダクトマネージャーPMMVPスコープ・ロードマップ整合・ユーザー価値
👤 ユーザー代表UXオンボーディング・エラー時UX・初心者〜上級者の連続性
⚙️ 技術アーキテクトARCH実現可能性・外部依存・将来拡張との矛盾
⚖️ 法務アドバイザーLEGALプライバシー・著作権・年齢制限・UGCリスク
📣 マーケターMKT差別化・SNS拡散・収益化との整合

② Phase 1b の強化（AC品質評価）

受け入れ基準（AC）を3つの軸で評価します。

評価軸内容
明確性機能要件（誰が・何をしたら・何が起きるか）として記述されているか。非機能要件は対象外。
完全性完全欠落・異常系欠落・複数操作欠落の3パターンを確認。仕様未確定は申し送りへ。
粒度（AC粒度）横断比較で著しく粗いACを検出。具体化案を複数提示しユーザーが選択。

③ 強化版 Question フォーマット

全フェーズ共通。背景・選択肢・影響を必ず明示し、回答者が判断しやすい構造にしています。

[Question 番号]
・問い   : 何を決める必要があるか（一文で）
・背景   : なぜこれが未決定なのか、なぜ問題になりうるか
・選択肢 : A案 / B案 /（不明な場合は「要調査」）
・影響   : 未決定のままだと〇〇フェーズの〇〇が進められない
[Answer 番号]

④ I/F 手戻り防止の連鎖（Phase 3〜6）

事前確認の問いがフェーズをまたいで引き継がれます。未解決の Question が残っている限り次フェーズへ進みません。各フェーズの確認はSTEP A/Bの2段階（調査→一覧提示→判断→確定）で実施します。

フェーズ生成する引き継ぎ文書主な確認内容
Phase 3if_precheck.mdデータ受け渡し・概念の異名・連携種別・エラー振る舞い
Phase 4if_precheck_p4.md例外系の扱い方針・順序制約・循環依存・PACT定義間の依存関係
Phase 5pact_index.md（未解決 Question 欄）フィールド仕様・異常系3種・べき等性・バージョニング
Phase 6domain_model_plan.md（I/F確認セクション）PACT接続コンポーネント・ACL担当・ドメインモデルの独立性

⑤ ナレッジログ（knowledge_log.md）

各フェーズ完了時に knowledge_log.md への追記を義務化。セッションをまたいで知見が蓄積されます。

フェーズ追記内容の例

フェーズ詳細

Phase 1 — ユーザーストーリー作成

熟練PMの役割で7ステップの計画を立ててからユーザーストーリーを作成します。ST4開始前にAIがエピック数・想定ストーリー数をもとに粒度（[A]エピック単位 / [B]ストーリー単位 / [C]一括）を推奨。[C]一括を選ぶ場合はPhase 1bでの確認コスト増について事前に説明があります。

Phase 1b — 整合性チェック

Phase 1 と同じセッションで実施。まずSTEP 1で全体調査、STEP 2でカテゴリ別問題一覧（矛盾・用語・AC品質・AC粒度・網羅性・申し送り）を影響度・推奨付きで提示。STEP 2.5でAC粒度の問題がある場合は具体化案を複数提示。後工程への申し送りは phase1b_handover.md に保存。

Phase 1c — アクターレビュー ★このプロンプト集の独自フェーズ

Phase 1 ST2 で定義したアクターを1人ずつ順番にレビューします。各アクターについて「行動の自然さ・目的との対応・抜け漏れ」の3観点で確認。各アクターのレビュー前に前フェーズ（Phase 1b）との整合性確認を実施。

💡 なぜアクターレビューが重要か

「ユーザーがこの操作を実際に行うか」「目的と合っているか」を設計着手前に確認することで、Phase 3以降でのユニット境界の設計根拠が明確になります。

Phase 1d — 専門家ペルソナレビュー ★オプション

起動時に5ペルソナの概要を提示し、使用するペルソナを選択します。「スキップ」でフェーズ全体をスキップできます。各ペルソナのレビュー前に前フェーズ（Phase 1c）との整合性確認を実施。

Phase 2 — 画面モック作成

STEP 0で3つの前提（作成物の種類・粒度・作業順序）を確認してから作業を開始します。フローチャート先行（[F]）を選んだ場合はSTEP 1aで網羅性チェック、STEP 1bで未決事項の精査（今決める/仮決めする/後回し可の3段階）を実施。inception_summary.md を参照して引き継ぎ事項を反映します。

Phase 3 — Unit of Work 分割

STEP 0で前工程成果物（inception_summary / actor_review / screen_flow）の影響項目を確認してから計画作成。各ユニットは概要（目的・主な責務・境界の根拠・他ユニットとの主な連携）→承認→ストーリー＋AC追記の2段階レビュー。I/F事前確認はSTEP A（要判断事項一覧提示）→STEP B（確定・保存）の2段階。

⚠️ if_precheck.md の全 Question 回答前にクリアしない

Phase 4 以降のI/F設計はこのファイルを基盤にします。未解決のまま進むと Phase 5 での PACT 設計で大きな手戻りが発生します。

Phase 4 — コンテキストマップ作成

STEP 1でコンテキストマップ（Mermaid記法）＋連携一覧テーブルを作成・承認、STEP 2でシーケンス図を作成・承認、STEP 3でI/F事前確認を2段階で実施。STEP 1の整合性確認でPACT定義間の依存関係を3パターン（A:ユニット境界問題/B:依存方向問題/C:正当な依存）に識別。パターンA・Bは解消してからPhase 5へ進みます。

Phase 5 — PACT 定義

計画作成時にAIが着手順序を提案（未解決事項のない連携を優先、共有ユニットを含む連携を連続着手推奨）。各連携はSTEP A（I/F確認チェックリストを確定済み/未確定の一覧で提示→承認）→STEP B（PACT定義→整合性確認）の2段階。STEP Bの整合性確認ではユニットの「主な責務」の範囲内かも確認。

Phase 6 — ドメインモデリング

STEP A（I/F確認：PACTとの接続点・ACL要否・独立性を一覧提示）→STEP B（計画作成）→STEP C（コンポーネント定義）の3段階。関連する全PACTファイル（pact_xxx.md）を参照。ACL担当コンポーネントの明示を計画・定義の両段階で確認。ユニットごとに繰り返し実行し、1ユニット完了ごとにセッションをクリアします。

生成物の品質確認「整合性確認レポートを提示してください」AIが自己チェックして問題を発見・修正してからレビューを求める
Question の記録（フォーマット通りに記載される）問い・背景・選択肢・影響が明示され判断しやすい

フェーズ再開保存済み md ファイルを貼り付けて「〇〇フェーズの△ステップから再開」どのステップからでも再開可能

トークンを節約するコツ

  • フェーズ完了後は必ずセッションクリアする
  • インセプションとコンストラクションでナビゲーターを使い分ける
  • Phase 6 は 1ユニット = 1セッション を原則にする
  • 不要なファイルを読み込ませない（必要なフェーズの md のみ参照）
  • Phase 1d（専門家ペルソナ）は必要なペルソナだけを選択する

前のフェーズへ戻る方法

各フェーズの成果物ファイルが手元にあれば、任意のフェーズから再開できます。新しいセッションを開き、対応するフェーズのプロンプトと保存済みファイルの内容を貼り付けて開始してください。

ナレッジログを活かす

プロジェクト完了後、knowledge_log.md に蓄積された知見を knowledge.md に転記して git push してください。次のプロジェクトからナビゲーターが自動で読み込み、同じ失敗を繰り返しません。
