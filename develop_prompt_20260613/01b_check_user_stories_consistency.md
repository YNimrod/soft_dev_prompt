【影響度の判定基準】
  高: Phase 3 以降（ユニット分割・PACT・ドメインモデル）で矛盾・手戻りを引き起こす
  中: Phase 3 以降で作業コストが上がるが致命的ではない
  低: 後工程への実害が少ない（読みやすさの問題など）

【推奨の基準】
  修正必須: 影響度「高」。このまま進むと後工程で手戻りが発生する
  修正推奨: 影響度「中」。修正しておくと後工程がスムーズになる
  任意    : 影響度「低」。修正しなくても後工程への実害は少ない

【[AC品質] カテゴリの判定基準】

  ■ 明確性（個々のACの記述）
  修正必須（高）: 機能として何をするかが全く書かれていない
    例: 「適切に動作すること」「正しく表示されること」「使いやすいこと」
    → 後工程で「何をもって完了か」が判断できなくなる

  修正推奨（中）: 機能の方向性は分かるが、主語・操作・結果のいずれかが欠けている
    例: 「商品が表示される」（誰が・どのような操作で・何が表示されるかが不明）
    → Phase 3 以降でユニット境界や設計の根拠が薄くなる

  任意（低）: 機能として意味は通じるが詳細化の余地がある
    例: 「ログイン後にダッシュボードが表示される」（内容の詳細は未定義だが機能は明確）
    → Phase 3 以降で詳細化できるため今は問わない

  ■ 完全性（ストーリーとACの対応関係）
  修正必須（高）: ストーリーの「何を」に対応する AC が 1 本もない
    例: 「ユーザーが商品を削除できる」というストーリーに AC が存在しない
    → Phase 3 でユニット定義ができない

  修正推奨（中）: 正常系 AC はあるが異常系が全くない、または
                  複数の操作を含むストーリーに 1 本しか AC がない
    例: 「検索できる」に正常系のみ → エラー時・0件時が未定義
    → Phase 5 の PACT 定義でエラー系の根拠がなくなる

  申し送り: 仕様として未確定のため Phase 1b では修正せず後工程へ引き渡す
    例: 「ページネーションの有無」「件数の上限」「キャンセル操作の有無」
    → [後工程への申し送り] カテゴリに記載し phase1b_handover.md に保存する

【[AC粒度] カテゴリの判定基準】
  対象: 個別評価（問題5・6）は通過するが、横断比較すると著しく粗いと判断できるAC

  具体化を強く推奨（同等ストーリーと比べて著しく差がある）:
    - 同エピック内の他ストーリーが操作・結果・例外を 3 項目以上定義しているのに
      このストーリーは 1 行のみ
    - 「〇〇できる」という一文のみで操作・結果・例外のいずれも記述がない

  具体化を推奨（方向性は分かるが詳細が不足）:
    - 正常系のみ記述されており、同等ストーリーでは異常系が書かれている
    - 複数の操作を含むと想定されるが AC が 1〜2 本のみ

  具体化案の提示方法（STEP 2.5 で使用）:
    - 案1（標準案）: 正常系＋代表的な異常系を追加
    - 案2（最小案）: 正常系の具体化のみ
    - 案3（詳細案）: 操作・結果・例外をすべて記述
    ユーザーが案を選択 or 修正 or 後回し（申し送り）を選択する
