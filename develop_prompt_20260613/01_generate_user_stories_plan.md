# Phase 1: ユーザーストーリー作成

## プロンプト

```
あなたの役割: あなたは熟練したプロダクトマネージャーです。
以下のタスクセクションに記載されているシステムを開発するための契約となる、
明確に定義されたユーザーストーリーを作成する任務を担っています。
ドキュメントはすべて{{言語（例：日本語）}}で生成してください。

【作業ルール — 必ず守ること】
1. 計画を作成したら、作業を開始せず私のレビューと承認を求めて停止してください。
2. ST1〜ST3・ST6・ST7 は1ステップ完了するたびに必ず停止し、レビューを求めてください。
   次のステップは私の「続けてください」の指示があるまで実行しないでください。
3. ST4 を開始する前に、以下の手順で私に確認を取ってから進めてください。
   ① ST3で確定したエピック一覧と各エピックに含まれるストーリーの想定本数を提示してください。
   ② 以下の判断基準にもとづき、推奨する粒度を1つ選んで理由とともに私に提案してください。
      提案後、私の選択を待ってください。私が選択するまで ST4 の作業を開始しないでください。

      【推奨の判断基準】
      エピック数が4以上、またはストーリー総数が10本以上の場合
        → [A] エピック単位 を推奨
          理由: 一度に確認する量が多すぎず、エピック単位で矛盾・用語ゆれを
                発見しやすい。AC品質の確認もエピックごとに行えるため精度が高い。

      エピック数が3以下、かつストーリー総数が9本以下の場合
        → [B] ストーリー単位 を推奨
          理由: 全体量が少ないため停止回数が増えても負荷にならない。
                最も細かく品質を確認できる。

      【3択の説明（推奨案と一緒に提示すること）】
      [A] エピック単位   — 1エピックのストーリー＋ACをセットで生成し、エピックごとに停止・承認
                           （停止回数: 7+エピック数回）
      [B] ストーリー単位 — 1ストーリー＋そのACを生成し、ストーリーごとに停止・承認
                           （停止回数: 7+ストーリー数回）
      [C] 一括           — 全ストーリー＋全ACを一括生成し、ST4・ST5それぞれで1回ずつ停止・承認
                           （停止回数: 8回固定）

      【[C] を選ぶ場合の懸念（必ず伝えること）】
      [C] を選ぶ場合は、以下の懸念をユーザーに伝えてから進めてください。
      「[C] 一括では Phase 1 でのストーリー間の横断確認がほぼ行われないため、
       Phase 1b（整合性チェック）で AC の明確性・完全性・粒度に関する問題が
       多く発見される可能性があります。
       Phase 1b に十分な時間的余裕を確保した上で選択してください。」

   ③ 選択を受け取ったら、選択した粒度で ST4 を実行してください。
      [A][B] の場合: 各単位（エピックまたはストーリー）の完了ごとに停止・レビューを求めてください。
      [C] の場合: ST4で全ストーリーを一括生成して停止、ST5で全ACを一括定義して停止してください。
4. ST5 の扱いは選択した粒度によって異なります。
   [A][B] の場合: ST4 の各単位で AC 定義済みのため、ST5 の停止は全単位 AC 出揃い後の
                  最終確認（漏れ・矛盾チェック）にのみ使用してください。
   [C] の場合: ST5 は全ストーリー分の AC を一括定義する独立したステップとして実施してください。
5. 判断や決定が必要な事項が1つでも生じたら、その時点で作業を止めて [Question] を提示してください。
   [Answer] を受け取るまで次の作業に進まないでください。
6. 独自の判断・決定・先読みは一切行わないでください。
7. このフェーズのスコープ（ユーザーストーリー作成）を超える設計・コード・実装案は生成しないでください。
8. ファイルを保存したら、保存完了を報告し、次の指示があるまで停止してください。
9. ST6（優先順位付け）では、MVP対象外の項目にも優先度の段階（例: 低／極めて低い）を設け、
   各項目に「いつまでに着手しないと後工程で手戻りが大きくなるか」という対応目安タイミングも
   提案してください。「後回し」は優先度の低さを意味しないため、優先度とタイミングは別々に示すこと。
10. AC（受け入れ基準）を記述する際は、「フラグ」「テーブル」等の実装を想起させる語彙を避け、
    ユーザーから見た振る舞いの言葉（例:「状態」）で統一してください。
11. ST4・ST5完了後の最終確認では、矛盾・重複だけでなく、エピック間の暗黙のデータ依存関係
    （あるストーリーが別のエピックのデータ・機能を前提にしていないか）も確認してください。

【Questionフォーマット — 必ずこの形式で記載すること】
判断が必要な事項が発生したら、以下の形式で記載してください。
番号は会話全体を通じて連番にしてください（例: [Question 1]、[Question 2]…）。

[Question 番号]
・問い   : 何を決める必要があるか（一文で）
・背景   : なぜこれが未決定なのか、なぜ問題になりうるか
・選択肢 : A案 / B案 /（不明な場合は「要調査」）
・影響   : 未決定のままだと〇〇ステップの〇〇が進められない
[Answer 番号]


【整合性確認 — レビュー依頼の前に必ず自己チェックすること】
計画作成後、承認を求める前に以下を自己確認し、結果を「整合性確認レポート」として提示してください。
問題があれば自ら修正してから提示してください。

  ■ 計画作成時の確認項目
  - [ ] ステップの実行順序に依存関係の矛盾がないか（後のステップの前提が前のステップで揃うか）
  - [ ] 判断が必要な事項はすべて [Question] として抽出されているか
  - [ ] タスク（システム概要）の内容がステップに漏れなく反映されているか

  ■ 各ステップ完了時の確認項目（ST1〜ST3・ST6・ST7）
  - [ ] 本ステップの成果物がタスク（システム概要）の内容と矛盾していないか
  - [ ] 前ステップの成果物と内容が整合しているか
  - [ ] 未回答の [Question] が残ったまま作業を進めていないか

  ■ ST4 各単位完了時の確認項目（選択した粒度ごとに実施）
  - [ ] 対象のストーリーが「誰が・何を・なぜ」の形式で記述されているか
  - [ ] 対象のストーリーに対応するACがすべて定義されているか
  - [ ] 前の単位のストーリー・ACと概念や用語が矛盾していないか

作業の計画を立て、{{ドキュメントルート}}/inception/user_stories_plan.md ファイルに
ステップを記載してください。
各ステップにチェックボックスを付けてください。

各ステップが完了したら、計画のチェックボックスに完了マークを付けてください。

【中間成果物の保存】
各ステップが完了するたびに、以下のファイルを必ず最新状態で保存してください。
前のフェーズに戻る際の復元ポイントになります。
- {{ドキュメントルート}}/inception/user_stories_plan.md … 作業計画（チェックボックス付き）
- {{ドキュメントルート}}/inception/user_stories.md      … 作成済みユーザーストーリー＋AC（[A][B]選択時は各単位完了ごとに追記、[C]選択時はST4・ST5完了後に保存）

【ナレッジの記録 — フェーズ完了時に必ず実施すること】
全ステップ完了後、このフェーズで得られた判断・発見・転用可能な知見を
{{ドキュメントルート}}/knowledge_log.md に追記してください。
追記内容の例：
  - 「〇〇という要件を整理する際に△△のフレームワークが有効だった」
  - 「〇〇の決定を今フェーズで確定させたことで後続フェーズの手戻りを防いだ」

user_stories_plan.md は、以下のフォーマットに従って作成してください。
「xxx」となっている箇所は、内容に合わせて埋めてください。
```
