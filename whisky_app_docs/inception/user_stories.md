## エピックA: ボトル登録・コレクション管理

| ID | ユーザーストーリー |
|---|---|
| A-1 | ウイスキー愛好家として、ボトルをカメラで撮影して登録したい。手動入力の手間をかけずにコレクションを記録できるようにするため。 |
| A-2 | ウイスキー愛好家として、ボトル情報を手動で入力して登録したい。画像認識でマスターDBに該当がない場合でも記録を残せるようにするため。 |
| A-3 | ウイスキー愛好家として、登録したボトルを「持っている」状態にしたい。自分の所有ボトルを一覧で把握できるようにするため。 |
| A-4 | ウイスキー愛好家として、登録したボトルを「飲んだことがある」状態にしたい。過去に飲んだ経験を記録として残せるようにするため。 |
| A-5 | ウイスキー愛好家として、登録したボトルを「飲みたい・買いたい」状態にしたい。今後欲しいボトルを忘れないようにするため。 |
| A-6 | ウイスキー愛好家として、「持っている」ボトルを未開栓／抜栓済みで区別したい。在庫の状態を正確に把握できるようにするため。 |
| A-7 | ウイスキー愛好家として、缶ハイボールを登録したい。瓶のボトルと同様にコレクション・記録の対象にしたいため。 |
| A-8 | ウイスキー愛好家として、一枚の写真から複数本のボトルを検出して登録したい。まとめて撮影した際の登録の手間を減らすため。 |

### 受け入れ基準（AC）

**A-1: カメラ撮影による登録**
- [ ] カメラでボトルを撮影すると、画像認識により銘柄が特定される
- [ ] メジャーブランドのボトルは高い精度で画像認識できる
- [ ] レアなボトル（ボトラーズ物・オールドボトル等）は認識に失敗する場合があり、その際はA-2（手動入力）にスムーズに引き継がれる
- [ ] 特定された銘柄がマスターDBに存在する場合、蒸留所・地域・国・度数・カスク種別が自動入力される
- [ ] マスターDBに該当がない場合、A-2（手動入力）のフローに引き継がれる

**A-2: 手動入力による登録**
- [ ] 蒸留所・地域・国・度数・カスク種別をユーザーが入力できる
- [ ] 各項目の入力を「保留」として選択できる（その場で確定させなくてよい）
- [ ] 保留にした項目があるボトルは、その旨が分かる状態で登録される
- [ ] マスターDBが後から更新され、保留にしていたボトルの情報に該当する更新がある場合、ユーザーに「情報を更新しますか？」と確認する
- [ ] 入力した内容（保留分を含む）でボトルが登録される

> 💡 **将来検討メモ**: マスターDB更新時の確認を「自動更新ON/OFF」で切り替えられるオプションを将来的に追加したい（今回はAC化せず、メモとして残す）

**A-3〜A-5: 3つの状態（持っている／飲んだことがある／飲みたい）**
- [ ] 「持っている」「飲んだことがある」は独立した状態で、ユーザーが個別にON/OFFできる
- [ ] どちらか一方をONにしても、他方が自動的に変化することはない
- [ ] 「飲みたい・買いたい」はユーザーがONにできるが、「持っている」がONになった時点で自動的にOFFになる
- [ ] 「飲みたい」と「飲んだことがある」は同時にONであってよい
- [ ] 各状態の一覧表示で、該当ボトルが表示される

> 💡 **設計フェーズへの申し送りメモ**: 「持っている」「飲んだことがある」「飲みたい」の状態遷移について、本ステップでは振る舞いレベルのルール（独立した状態・ウィッシュリストの自動解除）のみ確定した。状態遷移の詳細（例: 抜栓した瞬間の自動連動、複数の状態が絡む場合のUI上の表現等）は設計フェーズでより詳細に検討する。

**A-6: 未開栓／抜栓済みの区別**
- [ ] 「持っている」ボトルに対して、未開栓／抜栓済みのステータスを設定できる
- [ ] ステータスは後から変更できる

**A-7: 缶ハイボールの登録**
- [ ] 缶ハイボールをボトルと同様の方法（カメラ／手動）で登録できる
- [ ] 缶ハイボールも3つの状態（持っている／飲んだことがある／飲みたい）の対象になる

**A-8: 複数本検出**
- [ ] 一枚の写真に複数本のボトルが写っている場合、それぞれが個別の登録候補として検出される
- [ ] 検出された各ボトルについて、A-1と同様にマスターDB照合または手動入力に進める

## エピックB: テイスティング記録

| ID | ユーザーストーリー |
|---|---|
| B-1 | ウイスキー愛好家として、飲んだウイスキーについてテイスティングノートを記録したい。感じたことをその場で残せるようにするため。 |
| B-2 | ウイスキー愛好家として、記録したテイスティングノートを一覧で見たい。過去の記録を振り返れるようにするため。 |
| B-3 | ウイスキー愛好家として、テイスティングノートの詳細を見たい。当時感じた内容をしっかり振り返れるようにするため。 |
| B-4 | ウイスキー愛好家として、テイスティングノートを削除したい。誤って記録したものや不要になったものを取り消せるようにするため。 |

### 受け入れ基準（AC）

**B-1: テイスティングノートの記録**
- [ ] 記録は登録済みのボトル（エピックAで登録したもの）に紐づける。所有状況（持っている／飲んだことがある／飲みたい）にかかわらず紐づけ可能
- [ ] 1本のボトルに対して、複数回のテイスティング記録を作成できる
- [ ] 必須項目は「ひとことコメント」のみで、最小限の手間で記録を完了できる
- [ ] 詳細項目（評価軸・香り/味わい/フィニッシュ等のフェーズ別コメント）は任意で入力でき、後からでも追加・編集できる
- [ ] 記録した日付が自動的に保存される
- [ ] テイスティング記録を1件でも作成すると、そのボトルの「飲んだことがある」状態が自動的にONになる

**B-2: テイスティングノート一覧**
- [ ] 記録は一覧表示される。デフォルトの並び順は記録日の新しい順
- [ ] 以下の基準でソートできる: 記録日（新しい順／古い順）、ボトル名、蒸留所、地域、国、度数、カスク種別
- [ ] 一覧にはボトル名・記録日・ひとことコメントが表示される

**B-3: テイスティングノート詳細**
- [ ] 詳細画面では、記録時に入力したすべての項目（必須・任意問わず）が確認できる

**B-4: テイスティングノートの削除**
- [ ] 削除は完全削除（論理削除・復元機能なし）
- [ ] 削除前に「完全に削除され、復元できない」旨をユーザーに提示し、確認を得てから削除する
- [ ] 記録をすべて削除しても、ボトルの「飲んだことがある」状態はONのまま変わらない

## エピックC: SNS共有

| ID | ユーザーストーリー |
|---|---|
| C-1 | ウイスキー愛好家として、テイスティング記録を画像カードとして生成したい。SNSに投稿しやすい見栄えのいいビジュアルを得るため。 |
| C-2 | ウイスキー愛好家として、生成した画像カードを外部SNSへ共有したい。自分の記録をフレンドや他のユーザーに発信したいため。 |

### 受け入れ基準（AC）

**C-1: 画像カードの生成**
- [ ] カードには銘柄名・蒸留所／地域・評価内容（レーダーチャート等）・ひとことコメントが含まれる
- [ ] カードにアプリ名・ロゴが小さく表示される
- [ ] ダークモード／ライトモードの両方に対応する
- [ ] 縦長（9:16）と正方形（1:1）の2フォーマットから選べる

> 💡 **将来検討メモ**: カードに記載する内容（表示する評価項目の選択等）やカードサイズについて、将来的に複数のオプションから選べるようにしたい（テンプレート選択機能）。今回はAC化せず、固定フォーマット（縦長9:16／正方形1:1、内容は固定項目）で進める。

**C-2: 外部SNSへの共有**
- [ ] OS標準の共有機能（共有シート）を使って、任意のSNS・アプリへ画像を共有できる
- [ ] 特定のSNS（X等）への直接連携・自動投稿は行わない（まずは汎用共有で十分とする）

## エピックF: ヘルス機能

| ID | ユーザーストーリー |
|---|---|
| F-1 | ウイスキー愛好家として、飲んだアルコール量から体内から抜けるまでの予測時間を知りたい。次の予定や運転の計画に役立てるため。 |
| F-2 | ウイスキー愛好家として、飲んだスピリッツの摂取カロリーを知りたい。健康管理に役立てるため。 |
| F-3 | ウイスキー愛好家として、飲んだスピリッツと相性のいいフードのレコメンドを見たい。より良い飲み方を楽しむため。 |

### 受け入れ基準（AC）

**F-1: アルコール代謝予測**
- [ ] ヘルス機能内で、飲酒量・度数をその場で直接入力できる（エピックB・テイスティング記録とは独立した入力）
- [ ] 入力した飲酒量・度数から、アルコールが体内から抜けるまでの予測時間が表示される
- [ ] 予測値はあくまで目安である旨の免責表示を行う

**F-2: 摂取カロリー算出**
- [ ] ヘルス機能内で入力した飲酒量・度数から、摂取カロリーの目安が表示される

**F-3: フードペアリングレコメンド**
- [ ] ヘルス機能内で選択・入力したスピリッツの種類・特徴に応じて、相性のいいフードが提示される（テイスティング記録とは独立）

> 💡 **設計方針メモ**: エピックFはエピックB（テイスティング記録）と意図的に関連付けない。ユーザーがヘルス機能を使う際にその場で必要な情報（飲酒量・度数等）を入力する、独立した機能として設計する。

## エピックG: コミュニティ機能

| ID | ユーザーストーリー |
|---|---|
| G-1 | ウイスキー愛好家として、フレンドを登録したい。同じ趣味の人とつながれるようにするため。 |
| G-2 | ウイスキー愛好家として、自分の記録の公開範囲を設定したい。プライバシーをコントロールできるようにするため。 |
| G-3 | ウイスキー愛好家として、自分の在庫情報を共有したい。どのボトルがどれだけ持たれているか分かるようにするため。 |
| G-4 | ウイスキー愛好家として、好みが近いユーザーのレコメンドを見たい。新しいボトルとの出会いを増やすため。 |

### 受け入れ基準（AC）

**G-1: フレンド登録**
- [ ] QRコードまたは検索でフレンドを登録できる
- [ ] 検索可否はユーザーが設定できる

**G-2: 公開範囲設定**
- [ ] 記録の公開範囲を「匿名」「フレンドのみ」「全ユーザー」から選べる

**G-3: 在庫情報の共有**
- [ ] 自分の「持っている」ボトルの情報を、公開範囲設定に応じて他ユーザーに共有できる

**G-4: 好みベースのレコメンド**
- [ ] テイスティング記録の傾向から、好みが近いユーザーが飲んでいるボトルが提示される

> 💡 **申し送りメモ**: コミュニティ機能全般はバックエンド導入（Firebase / Supabase / 自前サーバー等）が前提となる（`03_dev_strategy.md`記載）。設計フェーズでバックエンド選定と合わせて検討する。

## エピックH: ゲーミフィケーション

> ⚠️ **優先度: 極めて低い**（他のMVP対象外項目よりもさらに優先度を下げる）

| ID | ユーザーストーリー |
|---|---|
| H-1 | ウイスキー愛好家として、テイスティングノートで記録できる内容がスキルツリーで段階的に解放されるようにしたい。使うほど深く記録できる楽しみを得るため。 |
| H-2 | ウイスキー愛好家として、マスターDBにないボトルを登録した際にポイントを獲得したい。新しい情報を提供するモチベーションを得るため。 |

### 受け入れ基準（AC）

**H-1: スキルツリーによる段階解放**
- [ ] 現在の対象ユーザー（中〜上級者）向けには、記録項目をほぼ全解放した状態で提供する（段階的な制限はかけない）
- [ ] 「初心者モード／上級者モード」のようなモード分岐による段階解放は、初心者層をターゲットに含める将来拡張（`system_overview.md`の将来展望）の段階で初めて実装する

> 💡 **将来検討メモ**: 初心者モードでの段階解放の詳細は、初心者層を対象に含めるタイミングで再検討する。現時点（中〜上級者のみ対象）では上級者向けの全解放のみを想定し、B-1（任意項目をいつでも追加・編集できる）との矛盾は発生しない。

**H-2: ポイント付与**
- [ ] マスターDBに未収録のボトルを手動登録すると、ポイントが付与される
