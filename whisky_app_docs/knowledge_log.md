# ナレッジログ（whisky_app プロジェクト）

## Phase 0（要件の具体化・明確化）

- 対象ユーザーを「初心者〜上級者まで全員」のように広く設定すると、MVPのコアジャーニーが1本に絞れなくなる。早い段階で「最も価値を感じてほしい一人/一種類のユーザー」を明示的に問うことで、機能の絞り込みがスムーズに進んだ。
- 既存の検討資料（開発戦略メモ等）がある場合、要件定義の対話と資料の記載内容に時期的なズレ・矛盾が生じやすい（例: 「前倒しでPhase2」と書かれていた機能が、対話の中でMVP必須に確定するケース）。整合性レビューのタイミングで両者を突き合わせ、差分を申し送り事項として明示しておくと後工程で迷わない。
- 「データの持ち方・クラス設計」と「やりたいこと（要件）」を意識的に分離してもらうことで、要件定義フェーズが設計の細部に引き込まれずに進められた。ユーザーから「設計段階で煮詰めればいい」という明確な切り分けの意思表示があった。

## Phase 0b（ゴールライン定義）

- 「ゴールラインに含めない＝後回し」は「優先度が低い」と同義ではない。今回、缶ハイボール対応・未開栓区別・複数本検出は優先度が高いにもかかわらずゴールライン（最低到達点）からは外れた。後回し項目には優先度ラベルを付けて整理すると、「後回し＝軽視」という誤解を防げる。
- カメラ画像認識のように「1本なら最小実装に含めるが、複数本検出は後回し」という機能の中での粒度分割が有効だった。機能全体を入れる/入れないの二択ではなく、機能内の難易度の高い部分だけを後回しにする判断が役立った。

## Phase 0c（デザインプリンシプル定義）

- 「対立軸」として提案したものが、対話の中で具体的な数値・基準（例: マスターDBを国内50本＋海外150本にする）まで決まってしまうと、それはもう「繰り返し使える判断ルール」ではなく「一回限りの決定事項」になる。design_principlesに残すべきかどうかは「今後似た対立が起きたときに、この内容自体を再利用できるか」で判別するとよい。再利用できない場合はgoal_line.md側に決定事項として記録する。
- 「シンプルさ vs 拡張性」のように提案した対立軸が、ユーザーの実感（項目を減らすことが本当に望むことではない）とズレている場合がある。「シンプルさ」という言葉が持つ含意（項目を削る）と、実際に解決したい問題（入力の手間を減らす）が一致しないことに気づき、軸自体を統合・再定義した。

## Phase 1（ユーザーストーリー作成）

- ゴールライン（最低到達点）とMVP全体は範囲が異なる。ストーリー作成のスコープを「ゴールラインのみ」にすると、後から高優先度の後回し項目（缶ハイボール対応等）を追加する際に手戻り・大改修のリスクが生じることに、ユーザー自身が気づいて指摘した。スコープ確定時は「最小実装」と「実装順序が後だが同サイクルに含むもの」を区別して提示すると、こうした手戻りリスクを早期に発見しやすい。
- AC作成中に「データの持ち方・排他性」のような設計レベルの話に踏み込みそうになった際、「フラグ」という実装的な言葉を使ったことが原因だった。「状態（ステータス）」という振る舞いレベルの言葉に統一するだけで、設計の話に広がらずに済んだ。AC作成時は実装を想起させる語彙（フラグ、テーブル等）を避け、ユーザーから見た振る舞いの言葉で統一するとよい。
- 複数エピックをまとめてストーリー化すると、エピック間の暗黙の依存関係（例: ヘルス機能がテイスティング記録のデータを前提にする）に気づきにくい。全エピック完了後の最終整合性チェックで、エピック間のデータ依存を明示的に洗い出すことが有効だった。
- 将来機能同士でも、依存関係をあえて切る（独立させる）という設計判断がある。ヘルス機能とテイスティング記録の関連付けを「あえてしない」という決定により、片方の変更が他方に影響しない設計にできた。
- 同じ「MVP対象外」でも優先度に段階がある（低 / 極めて低い）。ゲーミフィケーションのように、対象ユーザー層の前提（中〜上級者のみ）と機能の前提（初心者向け段階解放）が噛み合わない機能は、他の将来機能よりもさらに優先度を下げてよい。
